## 1. Architectural rules (PR blockers)

These come from `.claude/CLAUDE.md`, the "Принципы (нарушение = блокер PR)" section (lines 15-24) and the "Важные ограничения" section (lines 645-653).

PostgreSQL only, SQLite removed. The guide's principle 8 says PostgreSQL is the only mandatory dependency. The product has gone further than the guide text: SQLite has been removed from both the app and the test suite (MEMORY: v6.6.0 was the PostgreSQL-only cut). Ground truth in code: `backend/app/database.py` `create_engine_from_settings()` (line 201) is documented "PostgreSQL only. ... A non-PostgreSQL URL (empty or otherwise) raises so a misconfigured import fails loudly instead of silently falling back" (lines 204-207). No IfcOpenShell, no native IFC parsing. Constraints 1 and 3 (lines 647, 649): all BIM/CAD (DWG, DGN, RVT, IFC) goes through the DDC cad2data pipeline into the canonical JSON format. ## 3. Operational landmines (where past sessions lost work)

### 3.1 Embedded PostgreSQL survives process kills and silently falls back to SQLite
Source: `embedded_pg_recovery_timeout.md`, fix in `backend/app/core/embedded_pg.py` `boot()`. After a force-kill of the embedded postmaster, the next `serve` silently fell back to SQLite and stayed there. Health still reported `db=ok`, so it looked fine, but `DATABASE_SYNC_URL` was empty and the cost_db step died with `create_engine("")`. Root cause: pixeltable-pgserver hard-codes a 10s `pg_ctl start` timeout, which WAL crash-recovery exceeds, so `get_server()` raised even though the postmaster had already launched and was recovering in the background; `boot()` returned `False` before wiring the DB env vars. The fix retries `get_server` 6 times with a 4s sleep and evicts the half-built cached handle (`PostgresServer._instances.pop(...)`) between attempts.
Operational notes: the postmaster is started with `CREATE_NEW_PROCESS_GROUP`, so it survives a `TaskStop` of the python parent; leave it running and the new boot attaches fast. Get its port from the cmdline `-p <port>`. For dialect ground-truth look in the app log for `PostgreSQL tables` vs `SQLite tables` / `Context impl SQLiteImpl`, but do not trust a single grep. PowerShell `*>` writes logs as UTF-16-LE (BOM ff fe), so a UTF-8 Python read finds nothing - decode as `utf-16` (ripgrep/Grep auto-detect, so they still work). For numeric ground-truth immune to API scoping, connect psycopg2 straight to the port: `psycopg2.connect(host='127.0.0.1', port=<port>, user='postgres', dbname='postgres')`.

### 3.2 Worktree isolation pins a stale base ref - first action must be checkout main HEAD
Source: `feedback_worktree_isolation_stale_base.md`. Spawning an Agent with `isolation: worktree` can create a worktree pinned to a stale base ref, not current `main` HEAD. This bit 3 times in R6 and 5 more in the v4.8.0 wave. The symptom is silent: the agent does not error, it just writes business logic against missing tables/models, tests pass because the rest of that snapshot is also old, and the damage only surfaces on merge. Even an explicit "first instruction: git checkout SHA" can be defeated because the harness may provision at the parent HEAD-at-spawn-time, which moves forward during the run.
How to apply: the first instruction in any worktree agent must be: run `git log -1 --format=%H`; if not the expected SHA or a descendant, `git checkout <EXPECTED_SHA>` before doing anything. Add a work-specific sanity check (for example, verify file X contains class Y; if not, abort and report). Never trust the worktree subsystem to pick a fresh base.

### 3.3 NTFS junctioned node_modules cause a revert race in worktrees
Source: `feedback_worktree_junction_revert_race.md`. When a worktree's `frontend/node_modules` / `qa/node_modules` are NTFS-junctioned to the parent repo (the optimization that skips the multi-GB npm install), a worktree-side `git checkout` can write back to inodes shared with the main repo's working tree. The symptom is silent: an Edit reports success, the file looks right for a few seconds, then a worktree's `git checkout HEAD -- <path>` snaps it back and your edits vanish. NTFS junctions are inode-level links, not symlinks, so git on the worktree side does not know it is touching the parent tree, and the parent's `git status` then shows the file as clean. This cost the v4.8.0-to-v4.9.0 accommodation overlap fix.
How to apply: for files in modules that a running agent is touching, do not use Edit; write the complete file via one Bash heredoc and immediately `git add && git commit` in the same invocation so the race window is milliseconds. Better, wait for the verification wave to finish before touching shared backend modules. Diagnostic: if an Edit-then-Read shows the file at HEAD instead of your edit, and the file lives in a module an in-flight agent references, suspect this.

### 3.4 git push to main can silently no-op - verify with ls-remote and push by SHA
Source: `feedback_git_push_main_silent_fail.md`. `git push origin main` (or `main:main`) can report "Everything up-to-date" while origin/main is actually behind local HEAD, because the local `origin/main` tracking ref is stale (common right after fetching only a sub-branch) and git short-circuits. This caused a "Wave 1 merge not on GitHub" miss.
How to apply: after any merge commit you want on origin/main, verify with `git ls-remote <url> refs/heads/main`; if the remote SHA differs from local HEAD, push by explicit SHA refspec: `git push "https://${PAT}@github.com/.../OpenConstructionERP.git" $(git rev-parse HEAD):refs/heads/main`. That form bypasses the tracking-ref optimization.

### 3.5 Alembic agent migration fork - run alembic heads after any migration add
Source: `alembic_agent_migration_fork.md`. A subagent adding an alembic migration often chains it off whatever looked latest by FILENAME number rather than the true head, producing two heads, which makes `alembic upgrade head` ambiguous and blocks startup, CI, and VPS deploy. The embedded-PG `create_all` path hides it locally, so it goes unnoticed until a fresh alembic run.
How to apply: after ANY agent adds a migration, run from `backend` with a valid-shaped PG URL: `DATABASE_URL='postgresql+asyncpg://postgres:postgres@127.0.0.1:<port>/postgres' DATABASE_SYNC_URL='postgresql+psycopg2://...same...' python -m alembic heads` and confirm exactly ONE head. The valid PG URL is required because importing a migration triggers `app.database.create_engine_from_settings()`, which rejects non-PG URLs (`heads` itself does not connect). `alembic heads` is the only reliable head check; regex over `down_revision` reports false heads. To fix a fork, re-point the new migration's `down_revision` to the true head and renumber the file and revision id to sort after it (safe only when it is an unapplied leaf nothing references).

### 3.6 i18n single-quote duplicate-key blindness - run tsc and grep TS1117 after any translation pass
Source: `i18n_single_quote_dup_blindness.md`. The scratch i18n tooling (`_i18n_gap.mjs`, `_i18n_merge.mjs`) parses entries with a double-quote-only regex. Some locale files (ar.ts, ja.ts, pt.ts) carried hand-authored single-quoted blocks; the detector counted those keys as missing and the merge inserted double-quoted copies, creating 327 duplicate keys and 327 `error TS1117` (object literal cannot have duplicate properties). Duplicate keys fail `tsc --noEmit` (the CI / `npm run build` gate) and silently shadow translations at runtime (last-wins).
How to apply: after any locale-file translation pass, always run `cd frontend && npx tsc --noEmit | grep TS1117` (and the quote-agnostic `_i18n_dupcheck2.mjs`), not just the gap detector. Fix with `_i18n_dedup.mjs --apply`, which handles both quote styles, keeps the best non-English value per key, normalizes to double quotes, and removes the rest. Per-locale source files are the source of truth despite stale "do not edit" headers; `en.ts` is the i18next fallback.

### 3.7 Zombie background bash tasks can run a destructive git stash pop - kill the supervising bash first
Source: `zombie_bg_task_stash_pop.md`. An orphaned Claude background bash task from a prior session ran a probe with an unconditional `git stash pop` in an EXIT-trap `restore()`. Killing the visible child (a wedged pytest) made the linear probe advance, and because the tree was clean, the `git stash push` stashed nothing, so the unconditional pop yanked an unrelated old stash into the clean tree, producing 8 files with conflict markers and 58 dirtied files. A leftover background task has no port listener, so a port scan misses it; only `Get-CimInstance Win32_Process` for `python.exe`/`postgres.exe`/`bash.exe` plus CommandLine reveals it.
How to apply: before launching anything, enumerate `Win32_Process` for python/postgres/bash with CommandLine and flag orphans (parent PID gone, or a `shell-snapshots` bash). Kill the supervising bash FIRST with `Stop-Process -Force` (TerminateProcess skips bash EXIT-traps, so the stash pop never fires), then the child, then orphan PG. Run `git stash list` and `git status --porcelain` before and after. Recovery from a bad pop: `git reset --hard <published-HEAD-SHA>` preserves the stash stack and untracked files.

### 3.8 Frozen desktop builds crash on first DB connect (asyncpg SSL on loopback)
Source: `asyncpg_ssl_loopback_frozen.md`, fix verified present in `backend/app/database.py` at lines 241-263. The Windows/macOS/Linux desktop app "installed but did nothing on launch" (v6.8.0 and earlier) because the frozen backend died on its first DB connection. asyncpg defaults to sslmode "prefer", which eagerly builds an `ssl.SSLContext` while parsing connect args; in a frozen PyInstaller build the bundled OpenSSL cannot initialise default verify paths, so it raised `ssl.SSLError` and killed startup. Dev-from-source works (system OpenSSL), which is why it was invisible until installers shipped. The embedded PostgreSQL is plaintext on loopback and never negotiates TLS, so the handshake was both unnecessary and the thing that broke. Fix: in `create_engine_from_settings()`, detect a loopback host (`urlsplit(url).hostname` in "", "localhost", "127.0.0.1", "::1") and set `connect_args={"ssl": False}`; remote hosts keep asyncpg's default so external TLS PostgreSQL still works. Pair with the no-silent-fail launcher (logs to `~/.openestimate/desktop-launcher.log`, 240s) so the next such crash is visible. General lesson: any frozen build talking TLS through a bundled crypto lib can hit broken default verify paths; prefer turning TLS off where provably unneeded (loopback).
