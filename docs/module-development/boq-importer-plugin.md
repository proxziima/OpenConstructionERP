# BOQ Importer Plugin

OpenConstructionERP ships native importers for 20+ regional BOQ
standards (GAEB, NRM, MasterFormat, DIN 276, DPGF, PBC, GESN,
SINAPI, …) plus the cross-cutting GAEB XML 3.3 exchange. All of
them go through one polymorphic frontend page and one backend
dispatcher endpoint.

This guide explains how to add a 21st country — or override an
existing one — without writing a new React component or a new
backend module.

## What "BOQ importer plugin" means here

There are two kinds of plugins that touch BOQ import:

1. **Country pack** (this doc) — a registry entry that wires up
   a flag, label, column mapping, classification reference, and
   validator pack list for a specific national standard. No
   bespoke React or Python code is needed — the polymorphic page
   handles the UX, and the backend dispatcher routes to the right
   validation rule set.

2. **Native parser plugin** — a self-contained Python module that
   provides custom parsing for a non-CSV/XLSX format (FIEBDC-3
   BC3, GAEB XML, BSAB Excel macros, …). Mirrors the existing
   `ids_importer` pattern: drop a Python package under
   `backend/app/modules/<name>/`, give it a `manifest.py`, and
   the dispatcher discovers it on next boot.

The two layers compose: a country pack can reference a native
parser by id (`validatorPacks: ['bc3', 'pbc']`), and the
dispatcher picks the right parser based on magic-byte detection
of the uploaded file.

## Layer 1 — Adding a country pack (frontend)

A country pack is one TypeScript object plus one
`TradeSection[]` array.

### 1. Define the trade sections

In `frontend/src/modules/regional-exchange/data/tradeSections.ts`,
add a new `TradeSection[]` constant. The convention is
**code + bilingual label** (native + English in parentheses) so
the dropdown reads naturally to a country specialist while
remaining intelligible to a foreign reviewer:

```ts
/** Argentine SIRSO sections. */
export const AR_TRADE_SECTIONS: TradeSection[] = [
  { code: '01', label: 'Trabajos Preliminares (Preliminaries)' },
  { code: '02', label: 'Movimiento de Suelos (Earthworks)' },
  { code: '03', label: 'Cimentaciones (Foundations)' },
  // …
];
```

### 2. Register the country template

In `frontend/src/modules/regional-exchange/regionalRegistry.ts`,
append an entry to `COUNTRY_TEMPLATES`:

```ts
{
  id: 'ar-sirso',
  routeSlug: 'ar-boq-exchange',
  countryCode: 'AR',
  flag: '🇦🇷',
  label: 'Argentina SIRSO',
  formatHint: 'Excel / CSV in SIRSO format',
  importEndpoint: REGIONAL_IMPORT_ENDPOINT,
  validatorPacks: ['boq_quality', 'sirso'],
  excelTemplate: makeTemplate(
    'ar-sirso',
    'SIRSO',
    'Argentina',
    'AR',
    'ARS',
    '$',
    'SIRSO',
  ),
  tradeSections: AR_TRADE_SECTIONS,
  validateCode: (code) => /^\d{2}(\.\d{1,4})?$/.test(code.trim()),
  // sampleFile is optional — set it only if you ship a file in
  // frontend/public/templates/.
}
```

### 3. (Optional) Add a sample file

Drop a small but plausible sample at
`frontend/public/templates/ar-sirso-sample.csv`, then set
`sampleFile: '/templates/ar-sirso-sample.csv'` on the registry
entry. The polymorphic page shows a "Download a sample file to
try it" link when this field is set, and hides it when not.

Public-spec samples only. **Never** include a file generated by a
proprietary tool (Presto, Arquímedes, Bluebeam, …) — generate
samples from scratch matching the published specification of the
standard.

### 4. Done

That's all. The new entry:

- Mounts at `/ar-boq-exchange` automatically (back-compat slug).
- Appears in the command-palette / global search via the
  `searchEntries` synthesizer in `manifest.tsx`.
- Picks up the shared i18n bundle under the `regional.*`
  keyspace; no per-country translation file required.
- Routes file uploads to the existing
  `/v1/boq/boqs/{id}/import/auto/` dispatcher endpoint, which
  in turn picks the right country pack from the request body.

The two registry tests (`regionalRegistry.test.ts` and
`_registry.test.ts`) will need their fixture counts updated from
20 → 21.

## Layer 2 — Adding a native parser (backend)

Use this when the country pack needs to read a non-CSV/XLSX
format that requires custom parsing — like FIEBDC-3 BC3 (Spain),
GAEB DA XML (DACH), or BSAB-Excel macros (Sweden).

Mirrors the `ids_importer` pattern. The dispatcher endpoint
`/v1/boq/boqs/{id}/import/auto/` detects format via magic-bytes
+ extension and routes to your parser by id.

### Skeleton

```
backend/app/modules/boq_importers/
  manifest.py                # discovered by core/module_loader
  __init__.py                # on_startup() registers the parser
  parsers/
    ar_sirso.py              # def parse(file_bytes) -> ImportResult
    pl_zuzia.py              # … one file per format
  schemas.py                 # ImportResult / ImportPosition dataclasses
  router.py                  # exposes /import/auto/ dispatcher
  service.py                 # routes raw file → parser → ValidationContext
  validators/
    sirso_rules.py           # ValidationRule subclasses (optional)
  migrations/
    v0001_initial.py         # (no DB tables needed for parsers only)
```

### Parser interface

A parser is a single async function:

```python
from app.modules.boq_importers.schemas import ImportResult, ImportPosition

async def parse(file_bytes: bytes, *, filename: str) -> ImportResult:
    """Parse a single uploaded file and return canonical positions.

    The parser does NOT touch the database. Validation runs against
    the returned positions before they are persisted to oe_boq_position.
    """
    positions: list[ImportPosition] = []
    # … decode bytes (e.g. ISO-8859-1 for BC3), split records,
    # build ImportPosition objects, fill `classification` with the
    # standard's codes.
    return ImportResult(positions=positions, warnings=[], errors=[])
```

### Registering the parser

In `__init__.py`:

```python
from app.modules.boq_importers.service import register_parser
from app.modules.boq_importers.parsers.ar_sirso import parse as parse_sirso

async def on_startup() -> None:
    register_parser(
        format_id='sirso',
        extensions={'.sirso', '.xlsx'},
        magic_bytes=None,           # set if there's a sniffable header
        parser=parse_sirso,
    )
```

The dispatcher picks your parser when a request lands at
`/v1/boq/boqs/{id}/import/auto/` with either matching extension
or matching magic bytes. The matching country pack on the
frontend supplies the format id via `validatorPacks`.

### Validation rules

If your standard has its own classification regex / required
fields, register validation rules under
`backend/app/core/validation/rules/__init__.py` next to the other
rule sets (`din276`, `nrm`, `gaeb`, …). The rule id forms a pack
id you reference from `validatorPacks` in the frontend registry.

```python
@rule_registry.register
class SIRSOSectionCodeFormat(ValidationRule):
    rule_id = 'sirso.section_code_format'
    name = 'SIRSO section code must be NN or NN.NNN'
    standard = 'SIRSO'
    severity = Severity.ERROR
    category = 'structure'

    async def validate(self, context):
        # … check context.position.classification['sirso']
```

## Testing your plugin

### Country pack only

```bash
cd frontend
npm test -- src/modules/regional-exchange
```

The two existing test files cover the registry (`regionalRegistry.
test.ts`) and the polymorphic page rendering
(`RegionalExchangePage.test.tsx`). Add an extra case to each — see
the four design-mandated assertions for `es-pbc` / `it-computo` /
`uk-nrm` / `us-masterformat` for the pattern.

### Native parser

```bash
cd backend
pytest tests/modules/boq_importers/test_ar_sirso.py
```

Place a tiny but valid fixture file under
`backend/tests/fixtures/boq_importers/ar/` and assert that
`parse()` returns the expected `ImportPosition` list. Aim for at
least one passing position + one section/heading row + one
deliberately malformed row that should land in `warnings` or
`errors`.

## Anti-patterns

Things that look reasonable but cause grief downstream:

- **Don't** create a new `frontend/src/modules/<cc>-exchange/`
  directory. The whole point of Wave 5 Epic I was to collapse
  those into one. If you find yourself reaching for a copy-paste
  of the Spanish module, add a registry entry instead.
- **Don't** hard-code currency formatting in the parser.
  `ImportPosition.unit_rate` is a `Decimal`; let the BOQ view
  format it according to the project's currency setting.
- **Don't** include a sample file copied from a commercial tool.
  Build samples from the standard's published specification only.
- **Don't** call the BOQ position bulk endpoint
  (`/v1/boq/boqs/{id}/positions/bulk/`) from new code. The
  unified dispatcher at `/v1/boq/boqs/{id}/import/auto/` is the
  Wave 5 Epic I successor and runs validation@import.
