# RFC 37 — v2.6.0 Multi-currency, Editable IVA, Compound-position polish

**Status**: Draft · **Author**: Artem Boiko · **Date**: 2026-04-26
**Tracks**: GitHub Issues #88, #89, #93

## 1. Why

Three open issues from active users keep arriving at the same root: parts of the cost model that should be **per-project / per-line** are currently **locked to a regional template** or **omitted from the data model entirely**. The result is users — especially in Latin America — have to fight the tool to do basic estimation tasks.

| Issue | Pain                                                                                          | What user wants                              |
|-------|-----------------------------------------------------------------------------------------------|----------------------------------------------|
| #88   | Project base currency is the only currency. Foreign-priced materials force manual conversion. | Per-resource currency + project FX template. |
| #89   | VAT/IVA is set by `Project.region` and only editable post-BOQ-creation in the MarkupPanel.    | A `default_vat_rate` on Project itself.      |
| #93   | Compound positions: currency picker truncated, resource type stuck on "material", unit list closed. | Currency per resource, editable type, custom units. |

All three converge on three concrete data-model additions:

1. `Project.fx_rates`  — multi-currency template
2. `Project.default_vat_rate` — per-project VAT override
3. `Position.metadata.resources[i].currency` — already JSON, just needs UI + rollup logic

The compound-position polish (#93 items 2 & 3 — type editable, custom units) is pure frontend.

## 2. Goals — what ships

**v2.6.0 must do**:

- Project carries a list of additional currencies with FX rates to base; user can add/edit/delete via Project Settings.
- Project carries a `default_vat_rate` that takes priority over regional default at BOQ creation.
- Each resource on a position can declare its own currency; resource total in base = `qty × unit_rate × fx_rate`.
- Compound-position editor: resource type is a dropdown (material / labor / equipment / subcontractor / other), currency picker shows ALL project currencies including custom additions, unit field accepts free-form input that persists.
- Existing single-currency / regional-VAT projects keep working without migration steps — every new field is nullable / defaults to empty.

**v2.6.0 does NOT do** (deferred):

- Auto-fetching FX rates from external APIs (ECB / openexchangerates).
- Historical FX rate tracking (effective_date column). Rates are "current snapshot" only.
- Per-position currency override (only per-resource).
- Tenant-wide custom unit catalog (custom units stay project-scoped).

## 3. Data model

### 3.1 Project (new columns)

```python
fx_rates: Mapped[list] = mapped_column(
    JSON, nullable=False, default=list, server_default="[]",
    doc="Additional currencies + FX rate to project.currency. "
        "Shape: [{code: 'USD', rate: '1200.50', label: 'US Dollar'}]. "
        "Empty list means single-currency project.",
)
default_vat_rate: Mapped[str | None] = mapped_column(
    String(10), nullable=True,
    doc="Per-project VAT override as a percentage string (e.g. '21'). "
        "NULL means use the regional template's VAT row.",
)
```

Rate is stored as **string** (Decimal precision, SQLite parity).

### 3.2 Position metadata.resources[].currency (no schema change, JSON convention)

```json
{
  "resources": [
    {"name": "Cement bag", "type": "material", "unit": "bag",
     "quantity": 10, "unit_rate": 25.50, "currency": "USD"}
  ]
}
```

`currency` is **optional**. Absent → uses Project base currency. Backend rollup computes:
- `resource_total_in_currency = qty × unit_rate`
- `resource_total_in_base = resource_total_in_currency × fx_rate(currency → base)`

If `currency` is absent or matches base, fx_rate is 1.

### 3.3 Migration `v260c_project_fx_rates_vat`

Inspector-guarded ADD COLUMN for both fields, idempotent (matches the v260b pattern).

## 4. API surface

| Method | Path                                       | Purpose                                  |
|--------|--------------------------------------------|------------------------------------------|
| GET    | `/v1/projects/{id}`                        | Returns `fx_rates`, `default_vat_rate`.  |
| PATCH  | `/v1/projects/{id}`                        | Accepts both fields in update payload.   |
| PUT    | `/v1/projects/{id}/fx-rates/{currency}`    | Upsert one currency `{rate, label}`.     |
| DELETE | `/v1/projects/{id}/fx-rates/{currency}`    | Remove one entry.                        |

VAT override is a single field, no dedicated endpoint — PATCH the project.

BOQ creation reads Project.default_vat_rate before falling back to the regional template.

## 6. Compound-position polish (#93 items 2 & 3)

**Editable resource type**: Replace the read-only badge in `EditableResourceRow` with an inline select (`material/labor/equipment/subcontractor/other`). On change → `onUpdateResource(positionId, idx, "type", newValue)`.

**Custom units**: today `boqHelpers.ts` persists custom units in `localStorage`. Move to project metadata so units sync across browsers. Backend: a small `Project.custom_units: list[str]` column, exposed via the same project PATCH.

## 8. Risks

- Rate stored as string vs Decimal: covered — rates flow through `MoneyValue` which already does Decimal arithmetic.
- Old positions with no `currency` on resources: covered — absent treated as base.
- Mixing currencies in a single resource row: not allowed — each row has exactly one currency.
- Negative rates / non-numeric strings: validated at PATCH time (regex + Decimal parse).
- VAT override conflict with existing tax markup row: documented — on apply_defaults rerun, override wins.
