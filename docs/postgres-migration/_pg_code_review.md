# PostgreSQL Scale-Foundation — Read-Only Code Review

**VERDICT: 0 BLOCKERS, 3 HIGH**

The four newly-committed scale files (`pg_optimizations.py`, `database.py`, `costs/router.py` PG bulk-insert path, `migrate_sqlite_to_postgres.py`) are functionally sound and dialect-safe. No runtime PostgreSQL breaker remains in the application request path. The previously-suspected "json_extract" cluster in `bim_hub/repository.py` has ALREADY been migrated to the dialect-aware `json_path_text()` helper (confirmed at lines 25, 224, 232, 238, 244), so it is no longer a blocker. The remaining raw `json_extract` / `INSERT OR IGNORE` / `AUTOINCREMENT` / `datetime('now')` occurrences live exclusively in offline dev-only CLI scripts, in SQLite sidecar caches that open their own `sqlite3` connections, or behind an explicit `if conn.dialect.name == "postgresql"` split — never the app SQLAlchemy engine on the PG path. The three HIGH items below are real correctness/performance risks worth fixing before a production cutover, but none crashes on every install.

Scope reviewed: `backend/app/core/pg_optimizations.py`, `backend/app/database.py`, `backend/app/modules/costs/router.py` (dialect-aware CWICR bulk import), `backend/app/scripts/migrate_sqlite_to_postgres.py`, plus a cross-codebase grep for un-gated SQLite-isms.

## Severity-ranked findings

| Severity | file:line | Issue | Why it breaks on PostgreSQL | Suggested fix |
|---|---|---|---|---|
| **HIGH** | `backend/app/scripts/migrate_sqlite_to_postgres.py` `--truncate` path (reverse `sorted_tables` DELETE, no CASCADE / no `session_replication_role`) | Truncation relies purely on reverse-`metadata.sorted_tables` order to satisfy FK constraints; one `delete(table)` per table in a single transaction. | `sorted_tables` cannot topologically order a **cyclic** FK graph (self-referential `parent_id`, or mutual FKs between two tables). On PostgreSQL FKs are enforced immediately, so a `DELETE` on a table still referenced by a not-yet-deleted row raises `ForeignKeyViolation` and aborts the whole truncate transaction. SQLite (where this was likely validated) only enforces FKs when `PRAGMA foreign_keys=ON`, so the bug can be invisible in dev. Several app tables are self-referential (e.g. hierarchical BOQ positions). | For the PG truncate path, prefer one `TRUNCATE <all user tables> RESTART IDENTITY CASCADE` statement, or wrap the per-table deletes in `SET session_replication_role = replica;` ... `SET session_replication_role = DEFAULT;` to defer FK checks. |
| **HIGH** | `backend/app/core/pg_optimizations.py:127-134` (GIN index built with bare `postgresql_using='gin'`, no operator class) | GIN index on the JSONB `asset_info` / `classification` columns is created with the **default** opclass (`jsonb_ops`) rather than `jsonb_path_ops`. | Not a crash — it builds and works. But the only JSONB access the app issues is path/text extraction via `#>>` (through `json_path_text`); the default `jsonb_ops` index is ~2-3x larger and slower to build than `jsonb_path_ops`, and its extra capability (key-existence `?`/`?|`/`?&`) is never used. On the constrained 2GB VPS the oversized index is a real, avoidable cost. Also note: a bare `#>>` *text* extraction does not actually use a GIN index at all — only containment (`@>`) does — so verify whether this GIN index accelerates the actual asset-register query shape before investing in it. | Pass `postgresql_ops={col.name: "jsonb_path_ops"}` on the `Index(...)`. Confirm no code uses `?`/`?|`/`?&` key-existence operators first (none seen). If the asset filters need index acceleration, switch them to containment (`@>`). |
| **HIGH** | `backend/app/database.py:214-215` (`pool_size` / `max_overflow` applied unconditionally, SQLite included) | `pool_size` (default 24) and `max_overflow` (default 10) are set for **both** backends; the PG-only knobs `pool_pre_ping` / `pool_recycle` are correctly gated to `not _is_sqlite` (217-226). | Not a PG break in itself. Two concerns: (1) a 24+10 connection pool on single-writer SQLite invites `database is locked` contention (only masked by the `busy_timeout=30000` PRAGMA); (2) on PG, 34 max connections per uvicorn worker × N workers can exceed a small Postgres `max_connections` and cause `FATAL: too many connections` under load. | Decide deliberately: gate `pool_size`/`max_overflow` to the PG branch (or document the SQLite choice). For PG, ensure `(pool_size + max_overflow) × worker_count < server max_connections`. |
| MED | `backend/app/core/pg_optimizations.py:146-148` (all-tables fallback when `kw['tables']` absent) + `Base.metadata.create_all` on startup in `main.py` | The `after_create` event falls back to **every** table in the metadata when `tables` kw is missing; `main.py` runs `create_all` on startup. | Idempotent and safe (`idx.create(checkfirst=True)` per index, whole pass try-wrapped). But each pass issues an existence `SELECT` for every FK/composite/GIN index on every table — O(tables × indexes) catalog probes per boot on a 117-table schema. Startup tax + log noise, not a failure. | Return early when `kw.get("tables")` is falsy (a normal `create_all` always passes the just-created subset; the global fallback is what triggers the full re-scan). |
| MED | `backend/app/scripts/migrate_sqlite_to_postgres.py` `_coerce_row` (handles `str`→JSON only for generic `_JSON`-typed columns) | Columns declared directly as `sqlalchemy.dialects.postgresql.JSONB` in a model are NOT instances of the generic `JSON` for this isinstance check. | If any model uses `JSONB(...)` directly and a value arrives from SQLite as a JSON `str`, it is bound as a string into a JSONB column and PG rejects it. Generic-`JSON`→JSONB columns (the norm here, via the `@compiles` hook) are handled correctly. | Broaden to `isinstance(col.type, (_JSON, JSONB))` (`from sqlalchemy.dialects.postgresql import JSONB`). Grep models for direct `JSONB(` usage before cutover. |
| MED | `backend/app/database.py:198-206` (SQLite PRAGMA `connect` listener registered on the global `Engine` class inside `create_engine_from_settings`) | The listener is attached to the class-level `Engine`, not the specific engine instance, and registration happens inside a function. | Pure-PG deployments never register it (`_is_sqlite` is False), so it is safe in production. The risk is only a mixed-mode process (tests) where a SQLite engine is created first: the listener would then also fire `PRAGMA journal_mode=WAL` on a PostgreSQL dbapi connection and error. | Attach to the specific instance (`sa_event.listens_for(engine.sync_engine, "connect")` after creation), or guard the PRAGMA body with a per-connection dialect check. |
| LOW | `backend/app/scripts/migrate_sqlite_to_postgres.py` `_reset_sequences` (f-string interpolates `table.name` / `col.name` into raw `setval` SQL) | Identifiers are SQLAlchemy-derived (not user input) but interpolated unquoted. | No injection risk. Would only break on a reserved-word or mixed-case table/column name; the schema uses snake_case `oe_*` names, so safe in practice. Sequence reset correctly targets only integer PKs (`col.type.python_type is int`) and uses `pg_get_serial_sequence`, so GUID/string PKs are skipped (correct — they have no sequence). | Optionally quote identifiers via `quoted_name`. No action required for the current schema. |
| LOW | `backend/app/scripts/migrate_sqlite_to_postgres.py` docstring (stale module reference `app.core.performance_indexes`) | Docstring names a module that is actually `app.core.pg_optimizations`. | Documentation only. | Update the reference. |
| LOW | `backend/app/core/pg_optimizations.py:70-78` (`_index_name` 4-byte blake2b hash suffix only when name > 63 bytes) | Long index names are shortened with an 8-hex-char digest. | Collision is astronomically unlikely (32-bit digest over the full deterministic name) and `checkfirst=True` no-ops a true duplicate. Deterministic across re-runs. | None required. |

## Per-target-file confirmations (the reviewer's specific questions)

### `pg_optimizations.py`
- **JSON→JSONB `@compiles` hooks (53-64):** CORRECT and SAFE. Both the generic `sqlalchemy.JSON` and the dialect-adapted `postgresql.JSON` are caught, **both scoped to the `"postgresql"` dialect only** — SQLite DDL is never altered. They do NOT break the `json_serializer`/`json_deserializer` round-trip: SQLAlchemy keeps generic-JSON Python-side processing regardless of rendered column type, and `_tolerant_json_loads` tolerates psycopg returning an already-parsed dict (its `TypeError` branch catches `json.loads(dict)`). Native `postgresql.JSONB` columns are not touched (already JSONB) — fine, but see the migrate-script MED note.
- **`after_create` index event:** idempotent via `checkfirst=True`; GIN correctly skipped on non-PG (`is_gin and not is_pg`, 154); FK detection via `col.foreign_keys` correct; never aborts create_all (per-index try/except). The event is on `Base.metadata` and `main.py` runs a single `Base.metadata.create_all` (not per-module), so it is one pass per startup. Real concerns: all-tables re-scan (MED) and `jsonb_path_ops` opclass (HIGH).

### `database.py`
- `pool_pre_ping` / `pool_recycle` are PG-only (correct). `pool_size`/`max_overflow` on both backends (HIGH — deliberate decision needed). `GUID`, `_tolerant_json_loads`, slow-query listeners: portable, no PG issue. SQLite PRAGMA listener: PG-safe in production, MED only for mixed-mode test processes.

### `costs/router.py` — PG bulk-insert path
- `_pg_bulk_insert_cost_rows`: `create_engine(sync_url, pool_pre_ping=True)` → `with engine.begin()` (transactional) → `engine.dispose()` in `finally` — lifecycle CORRECT. `ON CONFLICT (code, region)` matches the real `UniqueConstraint("code", "region", name="uq_costs_code_region")` in `costs/models.py`, so it will NOT raise "no unique constraint matching". No SQL injection (column names hard-coded literals, values bound via `pg_insert().values()`). JSON object/array columns and `is_active` bool are decoded from str→Python before binding into JSONB. **Clean.**
- The SQLite raw `INSERT OR IGNORE` + write-tuning PRAGMAs are entirely inside the `use_sqlite = _is_sqlite(db_file)` branch and unchanged. **Clean.**

### `migrate_sqlite_to_postgres.py`
- Metadata discovery mirrors the alembic env; streamed batched copy in `metadata.sorted_tables` FK order; per-row fault isolation with retry passes for self-referential FKs; `_target_has_rows` guard; sequence reset PG-only and integer-PK-only. Real issues: `--truncate` cyclic-FK on PG (HIGH), direct-`JSONB` coercion gap (MED), stale docstring + unquoted identifiers (LOW). Naive→`timestamptz` on copy is low risk (0 timezone-aware columns; psycopg2 parses SQLite ISO strings into the `DateTime(timezone=True)` target).
